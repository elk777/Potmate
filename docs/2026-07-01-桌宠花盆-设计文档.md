## 0. 一句话内核

**用生活的碎片，浇灌一株只属于你的植物。**

一个跨平台（Windows / macOS）桌面桌宠：一只**空花盆**贴在桌面上。用户每天记录一件小事（喝了杯好咖啡、看到只肥猫），花盆里就会长出一株**由文字情绪决定的、独一无二的植物**。持续记录 = 浇水，植物生长；长期不记录，植物**慢慢枯萎**，但**补写就能救活，永不真死**。

## 1. 产品定位与目标

### 1.1 定位
- **品类**：情感陪伴型桌面桌宠 / 轻量生活记录工具
- **平台**：Windows + macOS（跨平台）
- **形态**：透明置顶的桌面小窗口，一只花盆常驻桌面
- **调性**：治愈、温柔、无压力、有惊喜

### 1.2 核心价值
1. **独一无二**：每株植物的叶形、颜色、花态由用户文字情绪当场生成，不可复制。
2. **情感投入**：植物会因你的疏忽而枯萎、因你的回归而复活，制造牵挂。
3. **无压力**：温和模式下植物永不真死，记录是享受而非负担。

### 1.3 非目标（明确不做的）
- 不做社交打卡 / 攀比排行（v1~v3）
- 不做重量级日记编辑器（记录就是一两句话）
- 不用"疯狂点击=健康"这种鼓励不健康行为的机制作为核心玩法

### 3.2 设计原则
- **核心领域逻辑纯 TS、无 UI 依赖**：植物生成、健康衰减、存储逻辑可脱离界面独立单元测试。
- **UI 是渲染层**：桌宠窗口只负责画和交互；领域逻辑只负责算。
- **可插拔的情绪分析**：本地规则与 AI 输出同一个 `EmotionProfile` 结构，上层无感知，v3 换 AI 不动上层。
- **小而聚焦的模块**：每个模块单一职责、接口清晰、可独立理解与测试。

## 4. 核心模块 1：植物参数化生成（灵魂）

### 4.1 输入 → 情绪特征（EmotionProfile）

```ts
// 输入：一天的文字记录 + 可选的手动心情 + 元数据
// 输出：
interface EmotionProfile {
  valence: number      // 情绪正负 (-1 ~ 1)  开心↔低落
  arousal: number      // 情绪强度 (0 ~ 1)   平静↔激动
  keywords: string[]   // 命中的意象词（猫/咖啡/雨…）
  length: number       // 文字长度归一化 (0 ~ 1)
  seed: number         // 由文本内容生成的稳定哈希
  moodSource: 'manual' | 'rule'  // 情绪基调来源：手动表情 / 规则推断
}
```

#### 诚实的能力边界

v1 的本地规则**不理解语义，只做模式匹配**——它是"关键词打分 + 规则映射"，能区分明显的开心/难过，但读不懂反讽、双关、复杂心情（例："忙到没时间难过"会被误判为负面）。真正"通人性"的语义理解要靠 AI（v3）。

因此 v1 采用**分工设计**，让每个信号只做它擅长的事：

| 信号来源 | 决定植物的什么 | 为什么 |
|---|---|---|
| **可选手动表情**（😍😊😐😔😢） | **情绪基调**（valence / arousal） | 用户亲手点的表情是 100% 准确的情绪信号，让最准的信号管最主观的东西 |
| **文字**（本地规则） | **意象装饰 + 独一无二的细节** | 从"咖啡/猫/雨"长出对应彩蛋、用内容哈希保证独特——这是文字分析做得好、且不需要"通人性"的部分 |

> **一句话**：文字决定"这株植物长什么样、有什么专属细节"，表情校准"它是暖色开心的还是冷色低落的"。

#### 情绪基调（valence / arousal）如何得出

1. **用户点了表情**（推荐路径）→ `moodSource='manual'`，直接由表情映射出 valence/arousal，**覆盖规则猜测**。这是精准、无挫败感的主路径。
2. **用户跳过表情** → `moodSource='rule'`，回退到本地规则推断：

```
情绪词典：
  正面词  开心/喜欢/温暖/治愈/舒服/惊喜…  → +分
  负面词  难过/累/焦虑/失望/孤独/烦…      → -分
程度词典：
  增强   非常/超级/特别/爆   → ×1.5
  减弱   有点/稍微/还行      → ×0.6
  否定   不/没/别            → 翻转符号

valence = Σ(命中情绪词 × 程度权重 × 否定符号) 归一化到 -1~1
arousal = 情绪词数量 + 感叹号数 + 程度词强度  归一化到 0~1
```

#### 意象与独特性（始终由文字决定，与情绪来源无关）

```
意象词典：
  动物  猫/狗/鸟   → keywords 增加 蝴蝶/瓢虫 装饰
  自然  雨/阳光/风 → keywords 增加 露珠/光斑 装饰
  食物  咖啡/蛋糕  → keywords 增加 特定小图标
seed = hash(文字内容 + 记录时间戳 + 记录序号)  // 见下方「seed 去重」
length = 文字长度归一化  // 越长 → 越繁茂
```

#### seed 去重（v1.1 修复：避免撞株）

> **问题**：纯 `hash(文字)` 会让「今天很开心」「累了」这类高频大众句必然生成**同一株植物**，当场击穿「独一无二」承诺。
>
> **修复**：seed 混入**记录时间戳 + 当前植物内的记录序号**，保证「同一句话在不同时刻/不同植物上，长出的细节不同」。
> - 纯文本 hash 仍单独保留为 `imageSeed`，只用于**意象彩蛋的可复现性**（同写「猫」总能召唤蝴蝶，这个「可复现」是想要的）。
> - 独特性 = `imageSeed`（意象可复现）× `uniqueSeed`（时刻不可复现）双 seed 分工。

- **意象词典比情绪词典更重要**：具体名词匹配准、惊喜感强，最能制造"这株植物是为我这句话长的"的感觉。
- **词典均为纯 JSON、可扩展、可热更**，后续能不断加词甚至社区贡献，无需改代码。

#### 输入退化与语言边界（v1.1 补齐）

| 退化输入 | 处理 |
|---|---|
| 空文字 / 纯 emoji / 纯链接 / 纯数字 | 判定为「无意象」，仍用时间戳 seed 长出一株**朴素但仍独特**的植物，不因内容贫乏而撞株 |
| 英文 / 中英混输 | v1 词典以中文为主，英文命中率低 → **意象退化但不报错**；v2 补英文词典（跨平台发版但情绪分析实质先只强中文，需在商店描述里诚实标注） |
| 超长文本（>200 字） | length 归一化封顶，避免繁茂度溢出 |
| 命中 0 情绪词且无手动表情 | valence=0 中性，走 §10「情绪分析异常」退化路径 |

> v3 用 AI 替换"情绪基调推断"这一层，输出同样的 `EmotionProfile` 结构，上层不改。届时手动表情保留为"AI 判完仍可微调"。

### 4.2 情绪 → 植物基因（PlantGenome）

```ts
interface PlantGenome {
  // ── 核心基因（出生首条记录锁定，之后不变 = 植物的「身份」） ──
  stemStyle: StemStyle        // 茎的曲直/分叉      ← 首记录 arousal
  leafShape: LeafShape        // 叶形(圆/尖/裂/心)  ← 首记录 seed + keywords
  leafColor: HSLColor         // 叶色(暖/冷/明/暗)  ← 首记录 valence
  flowerType: FlowerType      // 花形(无/单瓣/簇)   ← 首记录 valence（成长阶段解锁其显现）

  // ── 累积细节（第 2 天起每条记录追加，让「每日记录」持续有贡献） ──
  decorations: Decoration[]   // 彩蛋(蝴蝶/露珠…)   ← 每次记录 keywords 命中，累加
  density: number             // 繁茂度             ← 累积 length（越写越茂）
  accentShifts: HSLColor[]    // 局部色调点缀        ← 每次记录 valence，叠加为渐变层次
  flowerColor: HSLColor       // 花色（当前情绪）    ← 最近记录 valence + arousal
  swayAmplitude: number       // 摇摆幅度           ← 最近记录 arousal
}
```

#### 基因定型时机（v1.1 核心决策：核心定型 + 细节累积）

> 这是本设计最关键的一处修补——原 v1.0 未定义「植物的样子何时定型」，导致「独一无二」与「记忆连续」两个承诺无法同时成立。v1.1 明确采用**双层基因**模型：

```
出生 ── 首条记录 ──> 核心基因锁定（叶形 / 主色 / 茎形 / 花形 = 身份，永不变）
                       ↓
第 2 天起 ─ 每条记录 ──> 追加累积细节（蝴蝶/露珠/繁茂度/局部色调）
                       ↓
                    同一株植物：身份连续，但每天都在被你「添彩」
```

- **核心基因（身份层）**：首条记录一次性决定，锁定后不变。保证「这始终是我养的那一株」——满足 §5.3 的记忆连续。
- **累积细节（成长层）**：第 2 天起每条记录**追加**装饰、推高繁茂度、叠加局部色调。保证「每天记录都在真实地改变它」——记录的意义感成立。
- **两者兼得**：身份不变（连续性）× 细节日增（独一无二且每日有贡献），彻底化解 v1.0 的内在矛盾。

映射分工一目了然：**"情绪基调/身份"由首记录 valence/arousal 锁定（可被手动表情精准校准），"专属细节/繁茂"由后续每一条记录持续累积。**

- Canvas 用 `PlantGenome` + 程序化算法当场绘制：贝塞尔曲线画茎叶，可选 L-system 生成枝干分叉。
- **参数不同 → 结构真不同**，兑现"叶形/花形独一无二"。

#### 多记录合成（v1.1 补齐：一天/一生多条记录如何汇入基因）

> **问题**：一天可写多条、一株活一整段时间，`PlantGenome` 到底由哪些记录决定？v1.0 回避了。

- **核心层**：只认**该植物的第一条记录**（`birthRecord`），后续记录一律不改核心基因。
- **累积层**：
  - `decorations` / `density`：所有历史记录**并集/累加**（去重上限防溢出，如彩蛋种类封顶）。
  - `flowerColor` / `swayAmplitude`：取**最近一条记录**的情绪（表现「当下心情」，会随最新记录变化）。
  - `accentShifts`：滑动窗口（如最近 7 条）的 valence，叠成色调层次。
- 一天多条：**当天多条都计入累积层**，但成长度（§5.1）当天只推进一次（见 §10）。

### 4.3 渲染方式
- **纯参数化矢量（Canvas 2D）**，非预绘精灵图。
- 质感靠好配色 + 渐变 + 柔光 + 微妙纹理补足，走治愈系矢量插画风。
- v2 可选"关键部件叠贴图"增强质感（预留，非必须）。

## 5. 核心模块 2：健康模型（成长 · 活力双轴）

**两条独立的轴，互不干扰**（关键设计，避免"枯萎就缩回小苗"的 bug）：

### 5.1 轴① 成长度 Growth（决定"长多大"，基本只增）

```
种子 → 发芽 → 幼苗 → 成株 → 花苞 → 绽放
```
- 由**累积记录天数**驱动，每浇一次水推进一点，攒够进入下一形态。
- 表现"慢慢长大"。

### 5.2 轴② 活力 Vitality（决定"精神好不好"，会涨会跌）

```
满活力(100) ──不浇水随真实时间连续衰减──→ 枯萎阈值 ──→ 严重枯萎(下限, 不归零死亡)
              ↑ 补写记录 = 浇水, 活力回升 ↑
```
- **连续时间衰减**：桌宠常驻后台，活力按真实时间平滑下降（默认约 72 小时从满值降到枯萎阈值）。第 1 天就看得出没那么精神，全程可见、可挽回。
- **夜间/离机减缓衰减（v1.1 补齐）**：默认在**用户睡眠时段（如 23:00–07:00，可设置）与系统锁屏/休眠期间，衰减速率降至白天的 ~1/4**。否则用户每天早上打开都平白蔫一截，制造挫败而非牵挂。衰减「有效时长」按真实时间加权计算，非墙钟直减。
- **触底不死（温和模式，已确认）**：跌到下限停住，定格"严重枯萎"（土裂、叶垮、面色灰白），补写永远能救活。
- **枯萎有层次（v1.1 补齐）**：缺席时长不同，触底后的「枯萎表现」有梯度——缺席 3 天是「蔫」，缺席 1 个月是「叶落大半、只剩主干」，缺席 3 个月是「近乎枯枝 + 一点顽强绿意」。**表现分级但永不真死**，让「久别重逢」的时间感与「一浇就活」的治愈感都成立。
- **表现映射**：活力 → 低头角度 / 叶色枯黄度 / 动作速度 / 颜色饱和度。

### 5.2b 生命周期终点：绽放之后（v1.1 补齐）

> **问题**：v1.0 只说「养到绽放可移栽」，但没定义「养到绽放却不移栽」会怎样，缺少「圆满」的正反馈。

- 抵达「绽放」后进入**「盛放期」**：一段时间（如 7 天）内维持最美状态，是给用户的**情感高潮与奖励**。
- 盛放期内**主动、温柔地引导移栽**（气泡：「它已经开得很好了，要把它移进花园留作纪念吗？」），而非强制。
- 若用户始终不移栽：盛放期后进入**「结种」**——花谢、结出一颗种子彩蛋，可**一键把种子种回花盆开始新一株**（老株自动存入图鉴）。给「圆满谢幕」而非「永远悬停」。这条也解决了 v1.0「不移栽则图鉴永远空」的死角。

### 5.3 双轴分离的价值
一株已开花的成熟植物枯萎时，仍是那株大植物，只是"蔫了"；补水后原样复活，**记忆连续**，情感不断。

## 6. 核心模块 3：动画引擎（让它"活"）

一个**状态机 + 随机事件系统**。

| 动画层 | 内容 | v1 |
|---|---|---|
| **待机 Idle** | 呼吸 + 摇摆（simplex-noise 驱动，不机械） | ✅ 必做 |
| **随机小动作** | 偶发：叶子抖落露珠 / 蝴蝶飞过 / 打哈欠 / 扭头看鼠标 | ✅ 建可扩展事件库，先放 2-3 个 |
| **Hover 反馈** | 鼠标移上花朵：微微凑近 + 放大 + 柔光 | ✅ 必做 |
| **Click 反馈** | 戳一下：Q弹(squash & stretch) + 飘花瓣/星星 | ✅ 必做 |
| **浇水动画** | 提交记录瞬间：水壶浇水→土变深→植物精神一振→冒芽/绽放（**情感高潮点**） | ✅ 必做 |
| **枯萎/复活** | 活力变化驱动的下垂/返绿平滑过渡 | ✅ 必做 |
| 拖拽 | 拎起花盆在桌面拖动 + 落地弹跳 | 锦上添花 |
| 连点脾气 | 连续戳 → 转圈/头晕/背过身 | 锦上添花 |

- **GSAP** 管时间轴与补间，状态机决定当前该播哪条。
- 随机小动作做成**可扩展事件库**，后续能不断加新动作。

## 7. 核心模块 4：窗口与穿透（复用 OpenPet 方案）

### 7.1 桌宠窗口配置（tauri.conf.json）
```jsonc
{
  "label": "pet",
  "transparent": true,     // 透明
  "decorations": false,    // 无边框
  "alwaysOnTop": true,     // 置顶
  "skipTaskbar": true,     // 不进任务栏
  "shadow": false,         // 无阴影
  "resizable": false
}
// app 顶层：
"macOSPrivateApi": true    // macOS 透明窗口开关（必须）
```

### 7.2 点击穿透（CursorHitService）
定时器轮询（每几十毫秒）：
1. 取鼠标全局坐标 `cursorPosition()`
2. 取窗口位置 `innerPosition()` 与缩放比 `scaleFactor()`
3. 换算鼠标在窗口内坐标（除以 scaleFactor），判断是否压在花朵碰撞区
4. 压在花朵上 → `setIgnoreCursorEvents(false)`（可交互）；空白处 → `setIgnoreCursorEvents(true)`（穿透到桌面）

> 除以 `scaleFactor` 同时解决多显示器 / DPI 缩放问题。

#### CPU/耗电优化（v1.1 修正：轮询是中风险，非低风险）
轮询式全局坐标读取空闲时也持续跑，笔记本上耗电/发热会被用户感知。降耗策略：
- **区域自适应降频**：鼠标远离花盆时轮询降到 ~200ms，靠近时才升到 ~30ms。
- **窗口失焦/全屏应用前台时**暂停或大幅降频（用户在打游戏/看视频时不需要命中检测）。
- 动画与命中检测**共用一个 rAF/定时器主循环**，不各开一套。

### 7.2b macOS 辅助功能权限（v1.1 重大补齐）

> **被 v1.0 完全忽略的重大边界**：在 macOS 上读取**全局**鼠标坐标（`cursorPosition()`）属隐私敏感操作，很可能触发系统**「辅助功能 / 输入监控（Accessibility）」授权弹窗**。首次体验若冷不丁弹出吓人的系统授权，会劝退用户。

- **首启体验**：用一屏温和的「为什么需要这个权限」引导页（配桌宠插画）再唤起系统授权，而非裸弹。
- **降级方案**：若用户拒绝授权，退化为**「无穿透命中检测」模式**——桌宠整窗可点（点哪都能唤起记录窗），牺牲「空白穿透到桌面」但保证可用，不至于变砖。
- **实现前必须验证**：里程碑第 1 步的最小 demo 要在**真实 macOS 上确认**该权限的触发条件与降级路径（不同 Tauri/macOS 版本行为可能不同）。

### 7.3 其他
- **拖拽**：`startDragging()`（权限 `core:window:allow-start-dragging`）。
- **多窗口**：pet（透明）+ settings / record / gallery（普通）。
- **全屏 Space / Spaces（v1.1 补齐）**：macOS 全屏应用独占 Space 时，`alwaysOnTop` 的桌宠可能被遮或消失。策略：检测到前台为全屏应用时，桌宠可选择**跟随显示**（设 `NSWindowCollectionBehavior` 相关行为）或**优雅隐藏**并在退出全屏后归位，避免「桌宠凭空消失」的困惑。
- **位置越界回收（v1.1 补齐）**：显示器拔插/改分辨率后，保存的 `position` 可能落在屏幕外。启动时校验 `position` 是否在任一可见显示器内，越界则**吸附回最近可见边缘**，避免花盆「丢失」。
- **所需权限**（照抄 OpenPet capabilities）：
  `set-ignore-cursor-events` / `cursor-position` / `inner-position` / `scale-factor` / `start-dragging` / `set-position` / `set-size` / `show` / `hide` 等。

## 8. 数据结构与存储

```ts
// 一条记录
interface Record {
  id: string
  plantId: string         // v1.1：归属哪一株植物（多记录合成/图鉴关联的依据）
  date: string            // YYYY-MM-DD（按用户自定义「一天起点」归日，见 §10）
  text: string
  mood: MoodTag | null    // 可选的手动心情表情（用户跳过则为 null）
  emotion: EmotionProfile
  wateredAt: number       // 时间戳
}

// 手动心情表情
type MoodTag = 'joyful' | 'happy' | 'neutral' | 'down' | 'sad'  // 😍😊😐😔😢

// 当前植物状态
interface PlantState {
  id: string              // v1.1：植物唯一 id
  name: string            // v1.1：植物名（用户命名或按首条记录自动生成）
  genome: PlantGenome
  birthRecordId: string   // v1.1：核心基因来源的首条记录（定型锚点）
  growth: number          // 成长度
  vitality: number        // 活力
  stage: GrowthStage      // v1.1：显式成长阶段（种子…绽放…盛放…结种）
  birthDate: string
  lastWatered: number
}

// 图鉴：养成的植物存档
interface GalleryItem {
  id: string
  name: string            // v1.1：保留植物名
  genome: PlantGenome
  snapshotPath: string    // v1.1：卡片图存独立文件路径，不再内联 dataURL（见下）
  recordIds: string[]     // v1.1：只存 id 引用，记录本体单独存，避免重复膨胀
  period: { from: string; to: string }
}

// 全局设置
interface Settings {
  autostart: boolean
  position: { x: number; y: number }
  reminderTime: string | null
  dayStartHour: number    // v1.1：一天的起点小时（默认 4，凌晨前算前一天）
  quietHours: { from: string; to: string } | null  // v1.1：夜间减缓衰减时段
  deathMode: 'gentle'     // v1 固定温和模式
  theme: 'light' | 'dark' | 'auto'
  soundEnabled: boolean   // v1.1：音效开关
}
```
- v1 用 `plugin-store` 存**结构化数据**（记录/植物/设置）为本地 JSON。
- **快照图不进 JSON（v1.1 修正）**：`GalleryItem.snapshot` 若内联 base64 dataURL，图鉴一多会让 store 文件急剧膨胀、读写变慢。改为**存独立图片文件**（`plugin-fs`），store 只留 `snapshotPath`。
- **存储健壮性（v1.1 补齐）**：
  - 每次写入采用**临时文件 + 原子替换**，避免写一半崩溃导致 store 损坏。
  - 启动时校验 JSON，损坏则回退到最近一次**自动快照备份**（本地滚动保留最近 N 份）。
  - **本地导出/备份提前到 v1**（原计划 v5 太晚）：情感数据珍贵，用户丢了会心疼——至少提供「一键导出全部数据」。
- 图鉴数据量大后可迁移 SQLite（结构已预留）。

- 避免用户对着一个空花盆不知所措而流失。

## 10. 错误处理与边界

| 场景 | 处理 |
|---|---|
| 一天多次记录 | 以"当天是否浇过水"为准，多写追加进当天记录（都计入累积细节层），不重复推进成长 |
| **跨日定义（v1.1）** | 「一天」以 `Settings.dayStartHour`（默认凌晨 4 点）为界：4 点前记录算前一天，避免熬夜党「刚过 0 点就断签」的挫败 |
| 补写过去某天 | v1 简化：今天写了就算今天浇水；历史补写留存为记录但不倒推健康 |
| 系统休眠/关机 | 重启后按真实时间差一次性结算活力衰减；**结算时扣除夜间/锁屏减缓时段**（v1.1，见 §5.2） |
| 改系统时间 | 以记录时间戳为准，防作弊非 v1 重点 |
| Windows 透明穿透 | 动手第一步先跑"透明置顶+空白穿透"最小 demo 验证 |
| **macOS 辅助功能权限（v1.1）** | 全局鼠标坐标可能触发系统授权；先引导后唤起，拒绝则退化为整窗可点模式（见 §7.2b） |
| **窗口位置越界（v1.1）** | 显示器变更导致 position 落在屏幕外 → 启动时吸附回最近可见边缘（见 §7.3） |
| 情绪分析异常 | 退化为中性参数，仍长出一株普通植物，不崩溃 |
| **输入退化（v1.1）** | 空/纯 emoji/链接/纯数字/英文 → 意象退化但仍独特、不报错（见 §4.1） |
| 多显示器/DPI | CursorHitService 换算除以 scaleFactor |
| **store 损坏（v1.1）** | 原子写入 + 启动校验；损坏回退最近自动备份（见 §8） |
| **绽放后不移栽（v1.1）** | 盛放期后结种，可一键种回，老株入图鉴（见 §5.2b） |

### 版本演进原则
1. **灵魂优先**：v1 只做情感内核，把最难验证的"打不打动人"先跑通。
2. **风险后置**：AI、付费、后端、云等重资产推到 v3+，前期零后端、纯本地、纯前端。
3. **可插拔演进**：情绪分析（本地→AI）、存储（JSON→SQLite→云）都按接口设计，换实现不动上层。
4. **每版可独立发布**：任一版本都是一个完整可用的产品，而非半成品。

## 12. v1 实现里程碑（建议顺序）

1. **窗口地基**：Tauri + Vue 空壳，透明置顶穿透 demo 跑通（**最高风险先验证**）；**macOS 辅助功能权限的触发与降级同步在真机验证**（v1.1）
2. **植物生成**：EmotionProfile → PlantGenome（核心定型+细节累积）→ Canvas 静态渲染
2.5. **美术验证（v1.1 新增关键节点）**：静态渲染一出，立刻**找 5–8 名真人看「好不好看、想不想养」**。这是最高风险项，必须在大规模开发前先证伪/证真，而非等发版
3. **健康模型**：双轴逻辑 + 时间衰减 + 夜间减缓 + 枯萎分级（纯 TS + 单元测试）
4. **动画引擎**：呼吸 / hover / click / 浇水 / 枯萎 / 基础音效
5. **记录闭环**：记录窗口(shadcn-vue) → 浇水 → 植物变化 → 持久化(原子写入) → onboarding
6. **图鉴 + 命名 + 生命周期终点 + 导出备份 + 托盘/自启/提醒**
7. **打包发版**：tauri-action 双平台 CI

## 13. 已知技术风险与应对

| 风险 | 等级 | 应对 |
|---|---|---|
| **参数化植物"好不好看/感不感人"** | **高（v1.1 上调，最高风险）** | 这是开放式审美题、无明确完成标准，直接决定产品生死。里程碑第 2 步一出静态渲染就找 5–8 名真人验证，早失败早迭代 |
| **macOS 辅助功能权限（全局鼠标）** | **高（v1.1 新增）** | 可能触发吓人的系统授权、劝退首启；先引导后唤起 + 拒绝降级（§7.2b），里程碑第 1 步在真机验证 |
| Windows 透明穿透窗口 | 中 | 第一步先跑最小 demo；已知 OpenPet 方案可行 |
| **常驻内存/CPU 占用** | **中（v1.1 上调）** | 轮询式命中检测空闲也在跑，笔记本耗电可感知；区域自适应降频 + 全屏时暂停（§7.2） |
| 情绪分析"独一无二"感是否足够 | 中 | v1 靠双 seed + 意象词典拉开差异；已修复撞株漏洞（§4.1）；v3 上 AI 增强 |
| 多显示器/DPI | 低 | CursorHitService 已用 scaleFactor 换算 |

## 附录 A：关键决策记录

| 决策点 | 结论 | 理由 |
|---|---|---|
| 两种记录方案关系 | 方案1（文字）为主，方案2（键鼠）作辅助信号（v2） | 情感内核最强、隐私风险最低 |
| 文字→植物 | 文字管意象/独特性 + 可选手动表情管情绪基调；AI 增强留待 v3 | 本地规则不通人性，用表情做精准的情绪准星 |
| 植物渲染 | 纯参数化矢量（Canvas 2D） | 唯一能兑现"叶形/花形独一无二" |
| 健康模型 | 成长 / 活力 双轴分离 | 避免枯萎缩回小苗，保持记忆连续 |
| 枯萎到底 | 温和模式，永不真死 | 情感陪伴不该给负罪感 |
| 桌面框架 | Tauri 2.x | 轻量常驻、开源调性；坑已被参考项目趟平 |
| 前端框架 | Vue 3 + script setup + TS | 用户熟练、强类型 |
| UI 组件库 | shadcn-vue + Tailwind | 代码归己、可定制治愈风、不撞脸中后台 |
| 版本划分 | v1~v5 五个版本 | 主题清晰、风险递增、每版可独立发布 |
| **基因定型时机（v1.1）** | 核心基因出生定型 + 细节每日累积 | 同时兑现「独一无二/记忆连续」与「每日记录有意义」，化解 v1.0 内在矛盾 |
| **seed 策略（v1.1）** | 双 seed：意象可复现 × 时刻不可复现 | 纯文本 hash 会让大众句撞株，击穿「独一无二」 |
| **枯萎/终点（v1.1）** | 枯萎分级 + 绽放后盛放→结种→种回 | 抹平时间感、图鉴永空 两个死角 |
| **风险优先级（v1.1）** | 植物美观 = 最高风险，早做真人验证 | 审美是开放题无完成标准，比工程题更致命 |
| **macOS 权限（v1.1）** | 引导后唤起 + 拒绝降级为整窗可点 | 全局鼠标坐标可能触发辅助功能授权，裸弹劝退 |
| **备份/隐私（v1.1）** | 导出提前到 v1；AI 隐私前置到接入前 | 情感数据珍贵且敏感，不能事后补 |
